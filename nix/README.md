# Dashboard Kiosk NixOS Configuration

This directory contains the NixOS configuration for running a dual-browser kiosk dashboard.

## Features

- Auto-login as the `dashboard` user
- Sway window manager
- Launch two Chromium browsers in kiosk mode side-by-side
- Display https://cadmon.cob.org and https://bfd.firecat53.com
- Automatic monthly updates with reboot
- Foot, Git, Bottom and NetworkManager (nmtui) available for maintenance.
- SSH server enabled.
- Credentials
    * **Username:** dashboard
    * **Password:** dashboard

## Files

- `configuration.nix` - Production configuration for Raspberry Pi or similar hardware
- `hardware-configuration.nix` - Hardware-specific settings (generated by NixOS)
- `build-server.nix` - Reference configuration for the remote build host
- `vm-configuration.nix` - VM configuration for testing
- `build-vm.sh` - Script to build and run the test VM

## Keybindings

* Mod = Alt key
- `Mod+Return` - Launch terminal
- `Mod+e` - Restart the dashboard
- `Ctrl +` and `Ctrl -` - Increase and decrease browser font size as needed.

## Testing with VM

To test the kiosk configuration in a VM:

```bash
./build-vm.sh
./result/bin/run-nixos-vm
```

## SD Card Image (Flash and Boot)

Build a complete image you can write directly to a micro-SD card:

```bash
cd nix
nix build .#sdImage
```

The image is at `result/sd-image/*.img.zst`. Flash it:

```bash
zstd -d result/sd-image/*.img.zst --stdout | sudo dd of=/dev/sdX bs=4M status=progress conv=fsync
```

Replace `/dev/sdX` with your SD card device (check `lsblk`).

**First boot:** WiFi credentials aren't included in the image. Connect via
ethernet and SSH in, or attach a keyboard and run `nmtui` to configure WiFi.

**Build note:** Nix will substitute pre-built aarch64 packages directly from
cache.nixos.org — no cross-compilation needed. If your machine isn't already
configured to handle aarch64, add this to `/etc/nix/nix.conf`:

```
extra-platforms = aarch64-linux
```

Or on NixOS, add `boot.binfmt.emulatedSystems = [ "aarch64-linux" ]` to your
host config (which sets this automatically). In practice, all packages in this
config are in the cache, so the build is mostly just downloading.

## Production Deployment (existing Pi with NixOS)

```bash
git clone https://github.com/firecat53/bfd-apparatus ~/dashboard
sudo cp dashboard/nix/configuration.nix /etc/nixos/
sudo nixos-generate-config
sudo nixos-rebuild switch
```

## Remote Build Setup

The Pi offloads all compilation to a more powerful NixOS server. Monthly
auto-upgrades (`system.autoUpgrade`) use `--build-host` so the entire rebuild
happens on the server; only the final closure is copied back. Ad-hoc
`nix-build` / `nix-shell` commands also offload via `nix.distributedBuilds`.

### Files

- `configuration.nix` — Pi-side config (includes `nix.buildMachines` and
  `--build-host` flag)
- `build-server.nix` — Reference snippet to merge into the server's
  `/etc/nixos/configuration.nix`

### One-time setup

#### 1. Generate an SSH key on the Pi (as root)

```bash
sudo ssh-keygen -t ed25519 -f /root/.ssh/nix-build-key -N ""
```

#### 2. Configure the build server

Merge the options from `build-server.nix` into the server's
`/etc/nixos/configuration.nix`:

- **x86_64 server**: keep the `boot.binfmt.emulatedSystems` line so it can
  build `aarch64-linux` packages via QEMU.
- **aarch64 server**: remove the `boot.binfmt` line — it can build natively.
- Paste the Pi's public key (`/root/.ssh/nix-build-key.pub`) into the
  `nix-build` user's `authorizedKeys`.

Then rebuild:

```bash
sudo nixos-rebuild switch
```

#### 3. Update the Pi's configuration

Edit `configuration.nix` and replace `build-server` in `nix.buildMachines`
and `system.autoUpgrade.flags` with the server's IP address or resolvable
hostname. Adjust `maxJobs` to match the server's core count.

Then rebuild (this first build still runs locally):

```bash
sudo nixos-rebuild switch
```

#### 4. Test the connection

From the Pi as root:

```bash
ssh -i /root/.ssh/nix-build-key nix-build@build-server "nix store ping"
```

If this succeeds, remote builds are working. The next monthly auto-upgrade
will build entirely on the server.

### How it works

- `--build-host nix-build@build-server` in `system.autoUpgrade.flags` sends
  the full `nixos-rebuild` to the server via SSH. The server compiles
  everything, then the result is copied back to the Pi and activated.
- `nix.distributedBuilds` + `nix.buildMachines` handles any manual
  `nix-build` or `nix-shell` commands — the Nix scheduler offloads jobs to
  the server automatically.
- `nix.settings.builders-use-substitutes = true` lets the server pull
  pre-built binaries from `cache.nixos.org` directly, avoiding unnecessary
  compilation.

### Troubleshooting

- **SSH fails**: Ensure the Pi's root can reach the server
  (`ssh -i /root/.ssh/nix-build-key nix-build@build-server`). Check
  firewalls, hostnames, and that the public key is in the server's
  `authorizedKeys`.
- **Build still runs locally**: Verify `nix.buildMachines` hostname is
  correct and the server is reachable. Check `systemctl status
  nix-daemon.service` on both machines.
- **x86_64 server can't build aarch64**: Ensure `boot.binfmt.emulatedSystems`
  is set and the server has been rebuilt since adding it.
- **Disk space on Pi**: The garbage collector (`nix.gc`) runs weekly and keeps
  only the last 5 generations (`configurationLimit`). If the SD card fills up,
  run `sudo nix-collect-garbage -d` manually.
